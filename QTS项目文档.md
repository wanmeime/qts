## 一、项目概述

**QTS（Quantitative Trading System）** 是一个基于缠论技术分析体系的 A 股量化交易系统，目标是实现从市场扫描 → 热点发现 → 选股 → 买卖信号识别 → 自动交易的完整闭环。

- **工作目录**：`/home/jiaod/qts/`
- **核心方法论**：缠论 + MACD 辅助判断
- **部署环境**：WSL（Windows Subsystem for Linux），Docker 容器
- **交互方式**：通过飞书与 Claude Code 实时对话开发

## 二、用户交易系统与方法论

### 2.1 理论基础：缠论

缠论是一套科学解读市场的技术体系，核心理念是**不预测，只做完全分类**。主要概念包括：

| 概念 | 说明 |
|------|------|
| **顶分型/底分型** | 由三根K线构成，顶分型=上涨一笔可能结束，底分型=下跌一笔可能结束 |
| **笔** | 本级别最小单位，从底分型到顶分型（向上笔）或顶分型到底分型（向下笔） |
| **中枢** | 至少三笔重叠区间，构成走势的核心结构 |
| **趋势** | 两个同方向中枢构成趋势（上涨/下跌），时间跨度至少半年以上 |
| **买卖点** | 一买/一卖（趋势背驰点）、二买/二卖（回调确认点）、三买（回调不破前高） |
| **类二买** | 用户自创概念，在小级别横盘结束突破上沿时确认，力度最大速度最快 |
| **包含关系** | 相邻K线有包含时需合并处理，是缠论的基本操作 |
| **力度** | 离开中枢的一笔上涨/下跌的速度和角度，可用MACD辅助判断 |
| **背驰** | 走势力度减弱，预示转折，包括趋势背驰和盘整背驰 |
| **级别递归** | 日线→15分钟→分时图逐级精确判断买卖点 |

### 2.2 选股策略

1. **热门板块优先**：根据当天市场板块热点，找到领涨关注最高的板块
2. **缠论买点筛选**：结合缠论买点、卖点和中枢理论，对热门板块中的股票做走势分析
3. **排除亏损股**：亏损股票绝对不看、不选
4. **估值优先**：查看估值数据，优先选估值较低（历史低位、同行业偏低）的股票
5. **自选股池**：入选的股票加入自选，重点观察近期异动

### 2.3 买入操作

- **买入位置**：二买和类二买（右侧交易）
- **递归确认方式**：
  1. 日线出现底分型 → 观察是否确认
  2. 底分型确认 = 第四根K线收盘价 > 底分型第三根K线最高点
  3. 进入15分钟图，看小级别底分型和突破
  4. 进入分时图，精准抓突破买入时机
- **信号响应时间**：确认买入信号到执行操作控制在1分钟~半分钟内

### 2.4 卖出操作

- 卖出是买入的反向操作
- 第二类卖点 = 上涨回调后再次上涨未突破前期高点时确认

### 2.5 仓位管理

- 资金量不大，**全仓进全仓出**
- 除非涨幅已大且感觉风险，减仓一半

### 2.6 止损规则

1. **第一止损线**：二买的底分型最低点跌破，且没有快速收回 → 触发止损
2. **第二止损线（强止损）**：跌破一买的最低点（打穿前低）→ 无论损失多大必须止损
3. **买卖点推翻逻辑**：一旦价格打穿已确认的一买或二买的最低点，该买点即被证明构造失败，必须修正判断并触发止损

### 2.7 关键原则

- **不预测，只做完全分类**：当出现A情况做A操作，出现B情况做B操作
- **走势三种状态**：上涨、下跌、横盘（盘整）
- **中继分型概率**：第一次出现分型通常不会导致趋势反转，两次以上中继分型后，第三次反转概率大增
- **级别精确定义**：日线上一笔的转折点是比日线更小级别的买点，不是精确的日线级别买点
- **盘整不参与并非铁律**：类二买本质是在盘整即将结束时产生，是对缠论的改良

### 3.2 全市场扫描模块详细设计

**用户需求拆分**：
1. **指数扫描**：上证指数、深成指、创业板指数、沪深300、科创50 + 美股三大 + 恒生指数。每个指数只显示收盘点数+涨跌幅
2. **板块扫描**：行业板块+概念板块，指出当日领涨/领跌/异动板块
3. **三维评判**：
   - 事件驱动：偏空还是偏多
   - 情绪面：偏冷还是偏热
   - 资金面：偏流出还是偏流入
   → 综合判断市场整体是空还是多

**系统架构（8 Agent协作）**：
- 5位分析师并行：技术分析师（MA/RSI/MACD）、宏观分析师（板块轮动/资金流向/全球联动）、情绪分析师（涨停池/热门股）、新闻分析师（政策面/国际局势）、资金分析师（主力资金/板块资金流向）
- Bull vs Bear 辩论 2 轮
- Research Manager 综合 → 增强版报告

**报告格式**：HTML，发送到飞书群

### 3.3 缠论分析引擎设计

**分析逻辑（用户亲自描述）**：

```
1. 定位级别（日线为先）
2. 判断当前走势状态：
   a. 处于中枢中 or 中枢外
   b. 上涨一笔 or 下跌一笔
   c. 正在构造顶/底分型
   d. 相邻最近买卖点位置和价格
3. 分型判断：
   a. 新分型 → 中继分型计数
   b. 第一次中继通常不反转
   c. 两次以上 → 第三次反转概率高
4. 买点确认（递归方式）：
   a. 日线底分型出现 → 观察
   b. 确认 = 下一根K线收盘价 > 底分型高点
   c. 进入15分钟图精确判断
   d. 分时图精准抓突破
5. 买点推翻：
   a. 确认的买点被打穿 → 构造失败 → 触发止损
6. 操作信号：
   a. 出现买点 → 买入
   b. 出现卖点 → 卖出
   c. 无买点 → 观望
   d. 无卖点 → 持仓
```

**报告格式**：
1. **中期分析（日线）**：当前处于什么阶段（上涨/下跌/盘整），正在走向上的一笔还是向下的一笔，有无顶/底分型，分型是中继还是反转，最近买点/卖点位置
2. **短线分析（15分钟）**：15分钟图上的完整分析，同日线逻辑
3. **操作建议**：买入/卖出/持仓/观望

## 四、开发过程

### 4.1 Phase 1：市场扫描模块开发

**需求发现**：
- 用户提出需要整体市场分析模块
- Claude Code 搜索开源项目发现 TradingAgents 多 Agent 辩论框架
- 用户指出 Docker 里已部署两套 TradingAgents（15 Agent版 + 7 Agent版），经对比15 Agent更靠谱

**架构决策**：
- TradingAgents 市场分析师的能力可直接用于全市场扫描
- 复用 TradingAgents 第一层数据采集模块
- 在 TradingAgents 系统第二层增加全市场动态汇报功能板块

**开发方式**：
- 用户指示：单独建目录作为独立项目，调试完再集成到 QTS，避免污染主项目
- 在 Docker 容器中开发，集成到现有系统，QTS 通过 API 调用结果
- 使用 `qdrant` + `bge-m3` 向量数据库做知识库（Docker中有现成）

**迭代过程（报告）**：
| 版本 | 问题 | 用户反馈 |
|------|------|----------|
| v2 | 基础功能 | 缺全球指数，新闻事件联动不足 |
| v3 | 修复数据缺失 | 要找根源不要手动改 |
| v4 | 缺全球指数+维度评判 | 指出缺新闻事件、资金面/情绪面联动 |
| v5 | 补充维度表格 | 仍缺近期事件解读，指数太多 |
| v6-v8 | 多次修改 | 用户反复强调：必须有近期影响市场的具体事件 |

**关键修复**：
- 容器内 `market_insight_prompts.py` 的 PROMPTS 缺少4个key：`market_news_analyst_system/user`、`market_smart_money_analyst_system/user`
- 数据源问题：AKShare、东方财富、同花顺、新浪财经均可作为 A 股板块数据源
- TradingAgents 使用雅虎财经（yfinance）作为美股数据源

**定时任务配置**：
- 触发时间：每个交易日 A 股收盘后下午4点
- 任务内容：自动生成市场扫描报告
- 输出：HTML 格式发送到飞书群

### 4.2 Phase 2：缠论分析引擎开发

**知识库建设**：
- 搜索并下载缠论108课完整PDF（266页）
- 使用 `qdrant` + `bge-m3` 构建缠论知识库（实体连接）
- 多个子 Agent 并发切分处理页面（从93页中断处继续，分批并发处理）

**测试过程**：
1. 先测试基础知识（底分型、顶分型、笔的概念）→ 基本正确
2. 实测分析莲花控股 → 结果不满意（"胡说八道"）
3. 用户亲自描述缠论分析师应有的工作方式
4. 用户亲自描述完整分析逻辑 → 指导 Claude 从头重写
5. 再次实测 → 仍有多处错误（趋势定义、买点识别不对）
6. 用户认为 Agent 太浅，要求重新设计判断逻辑

**用户纠正的关键错误**：
- 缠论趋势定义：必须由两个同方向中枢构成，时间跨度至少半年
- 不应随意说"下跌趋势"，一笔下跌不是趋势
- 类二买的精确识别：小级别横盘结束突破上沿时确认
- 三买的定义：回调不破前高
- 买点推翻逻辑：打穿一买或二买的最低点 = 构造失败 = 止损

**交易系统整合需求**：
- 15分钟图分析是与日线相同的逻辑（级别递归）
- 15分钟数据显示：当前走势阶段、买卖点位置、操作建议
- 买卖价格用确认底分型的那根K线收盘价

### 4.3 Phase 3：Trading Agents 个股分析

**并行任务**：
1. 调用现有 Trading Agents 分析自选股100+只股票（每只都分析，输出汇总表格）
2. 开发缠论分析 Agent（独立开发，独立测试，最后集成）

**系统集成方案**：
- Trading Agents 的个股分析 + 缠论分析结合 → 更完整的分析视角
- 增加"缠论分析师 Agent"到 Trading Agents 系统 → 一步到位方案工程量大
- **分步实施**：先独立开发缠论 Agent，测试完成后再集成到 Docker 中的 Trading Agents

**美股规则问题**：
- Trading Agents 使用美股规则（支持卖空），对 A 股不适用
- Agent 说"做空入场"在 A 股无意义 → 需修改评判规则为 A 股逻辑

### 4.4 Phase 4：系统配置与调优

**LLM 模型更换**：
- Trading Agents 原用 MiMo V2/V2.5 → 用户要求更换为免费模型
- 用户计划提供新的 API Key 和调用接口

**时区问题**：
- 系统显示6月8日（周日），与实际时区不符
- 需调整 Trading Agents 时区设置为 UTC+8（北京时间）

## 六、问题与教训

### 6.1 重大事故：重建镜像覆盖旧版本

- Claude Code 从 TradingAgents-AShare 仓库重建 `tradingagents-final:latest`，覆盖了6月7日的旧镜像
- 旧镜像中的 market insight 功能全部丢失
- **教训**：**不要自作主张重建 Docker 镜像**，任何镜像操作前必须确认用户意图

### 6.2 语法错误反复出现

- 修改 `market_insight_prompts.py` 后出现语法错误，持续时间长
- 覆盖修改后丢失了新增功能，代码需要重写
- **教训**：用户要求加规则："当发现bug之后，一定要找出产生问题的根源，再修复。不要根据表面现象去修复"

### 6.3 主 Agent 崩溃问题

- 长时间运行的任务导致 Claude Code 主会话中断
- 用户要求：**每次执行任务都调用子 Agent，主 Agent 保持随时响应**
- 通过修改全局 Claude 配置文件（CLAUDE.md）实现该规则

### 6.4 A 股 vs 美股规则

- Trading Agents 默认使用美股交易规则（支持卖空）
- A 股没有卖空机制 → 分析结论完全错误
- 需要修改 Agent 评判规则适配 A 股

### 6.5 报告迭代教训

- 用户反复强调（5版以上）：报告中必须包含近期影响市场的**具体事件**
- 不能写"套话"——放到任何时候都适用的内容没有价值
- 新闻事件（如就业数据、美联储加息预期等）+ 市场联动解读是核心
- 指数列表精简：只要收盘点数和涨跌幅
